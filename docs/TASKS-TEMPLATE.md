# 任务清单 (TASKS)

> 使用方式：现场确定 `docs/SPEC.md` 后，复制本模板为 `docs/TASKS.md`。每个任务必须可独立执行、可验证、可交接。

## 任务状态说明
- [ ] 待开始 | [~] 进行中 | [x] 已完成 | [!] 阻塞

## 任务质量标准

每个任务必须包含：

- 任务编号和名称
- 优先级
- 依赖关系
- 涉及文件
- 输入和输出
- 验收标准
- 测试命令或验证方式

不合格任务示例：

- “完成前端”
- “完成后端”
- “优化体验”

合格任务示例：

- “实现 `/api/generate-listing`，接收 `ProductInput`，返回 `ListingGenerationResult`，AI失败时返回 mock 结果，验收为 curl 请求得到合法 JSON。”

## 时间轴概览

| 时间段 | 阶段 | 关键产出 |
|--------|------|----------|
| 0-1h | 基础设施 | 项目初始化、环境配置 |
| 1-7h | 核心功能 | API、AI集成、业务逻辑 |
| 7-11h | 前端界面 | 完整UI、交互体验 |
| 11-14h | 联调优化 | 端到端测试、性能优化 |
| 14-16h | 演示准备 | 部署、录屏、PPT |

## 风险项
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI API 调用限流 | 核心功能不可用 | 准备mock数据降级方案 |
| 第三方服务不稳定 | 功能缺失 | 本地缓存 + 重试机制 |
| 时间不足 | 功能不完整 | 严格按P0优先级砍需求 |
| 合规表述过度 | 误导评委和用户 | 只写“提示/缺口/待确认”，不写“已合规” |
| GitHub提交不足 | 影响提交和真实性证明 | 尽早初始化Git并保留开发过程 |
