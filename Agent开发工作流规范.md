# Agent 开发工作流规范

适用范围：本项目为跨境电商 AI 黑客松项目，采用 Qoder / OpenHarness 风格的规格驱动开发与多 Agent 协作流程。

核心目标：用前置规划、上下文管理和可验证交付，在有限时间内完成一个可运行、可展示、可解释的最小闭环 Demo。

## 1. 项目上下文

比赛主题是“AI + 电商：从痛点到方案”。项目必须围绕真实电商业务流程中的具体痛点，而不是泛泛地做“AI 赋能电商”概念展示。

项目原则：

- 不做大而全平台。
- 不堆功能清单。
- 优先完成一个可运行、可展示、可解释的最小闭环 Demo。
- 所有 AI 能力必须服务明确业务痛点。
- 每个功能都要能解释输入、处理过程和输出结果。

产品方向：

- 跨境电商开店注册引导工具（聚焦亚马逊平台）。
- AI 智能引导卖家完成开店注册流程。
- 材料校验 + 进度跟踪 + 常见问题解答。

明确边界：

- 不承诺自动完成注册，仅提供智能引导和材料校验。
- 不代替官方审核流程，仅帮助用户减少填写错误和材料缺失。
- 最终注册结果以亚马逊官方审核为准。

## 2. 比赛交付约束

提交物：

- PPT。
- GitHub 链接。
- 5 分钟以内录屏。
- 可运行 Demo 链接或本地演示方案。

评分结构：

- 技术实现：40 分，重点看 AI 是否真正参与核心流程、技术链路是否完整、Demo 是否稳定可运行。
- 创意 / 商业价值：30 分，重点看是否解决真实垂类电商痛点，是否有创新性和明确商业价值。
- 完成度：30 分，重点看核心功能是否完成、产品流程是否闭环、路演表达和材料是否完整。

一票否决：

- 核心功能非现场完成。
- 直接套用赛前完整项目。
- 未如实说明已有基础。
- 只展示概念、静态页面或 PPT，缺少真实可运行流程。

## 3. Qoder / AI Coding 工作流

Qoder 文档的核心不是“让 AI 多写代码”，而是让 AI 在清晰上下文、规格、任务边界和验证门禁中协作开发。

### 3.1 RepoWiki

RepoWiki 的目标是把隐性知识显性化，降低后续 Agent 和队友的理解成本。本项目的 RepoWiki 由以下文件组成：

- `README.md`：项目概览、技术栈、快速开始和提交物。
- `AGENTS.md`：项目级 Agent 工作规则。
- `MEMORY.md`：技术决策、当前状态和遗留问题。
- `对话.md`：用户沟通记录和处理过程。
- `HANDOFF.md`：跨 Agent / 跨会话交接。
- `docs/ARCHITECTURE.md`：通用全栈架构草案。
- `docs/SPEC-TEMPLATE.md`：需求规格模板。
- `docs/TASKS-TEMPLATE.md`：任务拆解模板。
- `docs/HACKATHON-BRIEF.md`：比赛手册摘要。
- `docs/QODER-AGENT-GUIDE.md`：Qoder 规范整理。
- `docs/PROJECT-BLUEPRINT.md`：默认项目蓝图。

### 3.2 Quest / SDD

复杂任务采用规格驱动开发：

```text
模糊需求
  -> SPEC.md
  -> PLAN.md
  -> TASKS.md
  -> 质量验证
  -> 单任务实现
  -> 测试
  -> Review
  -> 下一个任务
```

`SPEC.md` 负责回答“做什么、不做什么、为什么做”。

`PLAN.md` 负责回答“怎么做、按什么顺序做、风险如何兜底”。

`TASKS.md` 负责回答“谁做、改哪些文件、验收标准和测试命令是什么”。

禁止让 AI 一次性生成完整工程。复杂工程必须拆成可独立验收的任务。

### 3.3 前置规划比例

Qoder 文档强调复杂任务要把 40%-60% 时间放在需求澄清和计划上。黑客松场景下，这不是拖慢进度，而是减少返工。

必须优先回答：

- 用户是谁？
- 真实痛点是什么？
- 输入是什么？
- AI 做了什么不可替代的处理？
- 输出如何直接用于电商工作流？
- 5 分钟演示如何讲清楚？
- 哪些功能是 P0，哪些必须砍到 P1/P2？

## 4. 多 Agent 分工

主控 Agent 是 Coordinator，负责总决策和质量把关。子 Agent 只承担明确子任务。

### 4.1 Coordinator

职责：

- 汇总 Search / Explore / Plan / Worker / Verification / Evolution 的结果。
- 裁决冲突意见。
- 控制范围，防止需求膨胀。
- 保证交付物满足比赛规则。
- 维护 `MEMORY.md`、`对话.md` 和 `HANDOFF.md`。

### 4.2 Search Agent

职责：

- 调研跨境电商真实痛点、竞品、公开 API、开源方案和平台约束。
- 优先查官方、一手和可信行业资料。
- 输出可转化为产品/架构决策的结论。

本轮 Search 结论：

- 调研结论：亚马逊开店注册流程（资质准备→账户创建→店铺设置→审核通过）。
- 核心痛点：材料准备复杂、填写易出错、审核驳回率高。
- AI 切入点：智能引导 + 材料校验 + 进度跟踪。
- 该方向适合 24 小时内做出最小闭环 Demo。

### 4.3 Explore Agent

职责：

- 只读检查当前代码库、文档、配置和运行状态。
- 确认是否为 Git 仓库。
- 标出可复用文件和缺口。

本轮 Explore 结论：

- 已有 `README.md`、`docs/ARCHITECTURE.md`、`docs/SPEC-TEMPLATE.md`、`docs/TASKS-TEMPLATE.md`。
- 当前目录不是 Git 仓库。
- 原本缺少 `AGENTS.md`、`MEMORY.md`、`对话.md`、`HANDOFF.md`，本轮已补齐。
- 现有架构偏通用 Next.js 黑客松模板，需要用跨境电商小闭环收束。

### 4.4 Plan Agent

职责：

- 将需求拆成 SPEC、PLAN、TASKS。
- 明确 P0/P1/P2。
- 设计 Demo 路径。
- 给出任务依赖、涉及文件、验收标准和测试命令。

任务质量要求：

- 每个任务必须能被单独领取。
- 每个任务必须有明确验收条件。
- 不允许出现“完成前端”“完成后端”这种粗任务。
- 不允许把核心功能放进“后续完善”。

### 4.5 Worker Agent

职责：

- 按 `TASKS.md` 完成代码、文档或演示素材。
- 只改自己负责的文件。
- 不覆盖用户或其他 Agent 的改动。
- 不把 mock 结果伪装成真实 API 结果。

### 4.6 Verification Agent

职责：

- 运行构建、测试、lint 或最小启动验证。
- 检查 Demo 是否能按 5 分钟路径跑通。
- 检查 AI 输出是否结构化、可解释、可复现、有降级。
- 检查合规提示是否明确为“风险提示 / 缺口提示”，不是法律结论。

文档任务的验证要求：

- 文件存在。
- 关键比赛规则没有遗漏。
- Qoder 25 页核心方法有被转化为本项目工作流。
- 下一个 Agent 能直接读文档继续工作。

### 4.7 Evolution Agent

职责：

- 清理无用临时文件。
- 更新 `MEMORY.md` 和 `对话.md`。
- 必要时更新 `AGENTS.md` 和 `HANDOFF.md`。
- 记录踩坑和下一步。

Evolution 只写有复用价值的信息：决策、状态、验证、风险、遗留问题。

## 5. 上下文管理规范

每轮任务开始必须确认：

- 是否存在 `AGENTS.md`。
- 是否存在 `MEMORY.md`。
- 是否存在 `对话.md`。
- 是否存在 `HANDOFF.md`。
- 是否为 Git 仓库。
- 当前任务应读取哪些 `docs/` 文件。

上下文投喂原则：

- 优先引用具体文件，不让 Agent 盲搜。
- PDF 和长文档要先提炼成项目规则，再给 Worker 执行。
- 上下文占用超过约 60% 时拆分任务。
- 子 Agent 只拿完成任务需要的上下文。

## 6. Rules / Memory / Skill / MCP / Hooks

Rules：

- `AGENTS.md` 是项目级规则，约束所有 Agent。
- 不能硬编码密钥。
- 不能虚构实现。
- 不能承诺自动合规。
- 不能扩大到大而全平台。

Memory：

- `MEMORY.md` 记录技术决策、状态、验证证据和遗留问题。
- 每次任务结束前必须更新。

Skill：

- 固定流程可以沉淀为 Skill，例如 PDF 摘要、Web 调研、文档验证、Demo 录屏检查。

MCP：

- 用于接入浏览器、数据库、GitHub、Figma、外部 API 等工具。
- 黑客松 MVP 中按需使用，不为了展示概念而引入。

Hooks：

- 可用于阻断危险命令、检测密钥、写后自动 lint、任务结束提醒。
- MVP 前期可以先用人工检查，后续再自动化。

## 7. Demo 最小闭环定义

最小闭环必须包含：

```text
输入 -> AI 处理 -> 结构化输出 -> 可演示业务价值
```

默认 Demo 路径：

1. 打开首页，看到“亚马逊开店注册引导”入口。
2. 选择开店类型（个人/企业）和目标站点（美国/欧洲/日本等）。
3. AI 生成个性化注册流程清单和所需材料列表。
4. 用户上传/填写材料，AI 实时校验并给出修改建议。
5. 展示注册进度跟踪面板，明确当前步骤和待完成项。
6. AI 解答注册过程中的常见问题和注意事项。
7. 展示最终注册检查清单，确保所有材料就绪可提交。

## 8. 风险与降级

AI API 不稳定：

- 准备 mock 结果。
- 页面明确提示“演示数据”。
- 保证录屏和现场演示不断链。

时间不足：

- 只保留单页主流程。
- 放弃登录、数据库、多用户、复杂历史记录。
- Supabase、图片识别、多平台发布都可降为 P1/P2。

合规风险：

- 避免“自动合规”“保证通过审核”等表述。
- 使用“可能缺少”“建议补充”“需人工确认”。

非 Git 仓库：

- 正式开发前建议初始化 Git。
- 比赛需要 GitHub 链接，应尽早推送，保留开发过程记录。

## 9. 每轮结束清单

每次任务结束前必须确认：

- 用户原始需求已记录到 `对话.md`。
- 技术决策和状态已记录到 `MEMORY.md`。
- 若有后续 Agent，`HANDOFF.md` 已更新。
- 验证步骤已执行或明确说明无法执行的原因。
- 遗留问题和下一步明确。
